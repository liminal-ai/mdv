  mizing the notarization workflow
  (https://developer.apple.com/documentation/security/customizing-the-notarization-workflow).
  6. Staple the notarization ticket.
     That bakes the notarization proof onto the distributed app/package so Gatekeeper doesn’t have to look
  it up online every time.
  7. Verify locally.
     Typical checks are with codesign, spctl, and notarization logs before you distribute.

  - making sure helpers/frameworks are signed correctly

  ## Web-based directory browser

  Yes, this is very doable, and it can work for Electron too.

  The key question is where filesystem access lives.

  ### Best approach

  Put directory browsing in the local Node server.

  Then both:

  - browser app
  - Electron app

  use the same API like:

  - GET /api/tree?root=/Users/...
  - GET /api/file?path=...
  - POST /api/save
  - POST /api/export/pdf
  - etc.

  That means the directory browser is not “browser-native”; it is API-driven, which is exactly what you want
  if both Electron and browser should share it.

  ### Why this is good

  - one implementation of folder traversal
  - one implementation of file watching
  - one implementation of export

  expected. It is a local macOS markdown viewer/editor/export tool with Mermaid support, a file tree,
  multiple document tabs, light editing, and export to PDF, DOCX, and HTML. It began as a fast prototype,
  but after spending time with it, the conclusion was that it is not throwaway code. It is a legitimate
  first-pass proof of concept with enough real value to use as a product seed. The current app is being
  actively used locally on desktop, so an important constraint is that it needs to remain runnable and
  maintainable while we plan the future version.

  We spent time reviewing the existing codebase from a design, maintainability, and performance perspective.
  The high-level conclusion was that the app has a solid macro split between reusable core rendering/export
  logic, Electron shell code, and renderer UI code, but it had accumulated oversized orchestration files and
  some duplicated IPC surface. A major cleanup/refactor pass was completed to improve that. The current
  implementation was restructured so the Electron app is much more modular: shared IPC contracts were
  introduced, the main process was broken into services and bootstrap code, and the renderer was split into
  bridge, state, view, and controller layers. Tests were expanded and the app still builds and passes the
  suite. So where we are now is: we have a working, refactored Electron prototype that is stable enough to
  keep using as a reference and fallback.

  At the repo level, we decided to stop treating the current Electron implementation as “the app” in the
  long-term sense. Instead, it is now explicitly a prototype. The current Electron app has been moved into
  first-pass-poc/, and that folder is now the canonical place to build and run the current desktop version.
  The root of the repo is now intended to be more of a planning and future-build workspace. The repo also
  has docs/spec-build/01--preliminary/, which contains the first architecture and product planning docs, and
  an app/ folder reserved for the next-generation implementation. The key point here is that the current
  Electron app remains usable and important, but it is no longer the assumed center of the long-term product
  architecture.

  The next major topic we worked through was what the next-generation product should actually be. We
  considered several directions: continuing to deepen the Electron app, moving to Swift/SwiftUI, turning it
  into a local web app with a Node backend, and even different stack choices for that local web app. The
  main conclusion was that the best long-term direction is a browser-first, local-first application rather
  than an Electron-first or native-first one. The reason was practical, not ideological. The app is meant to
  be easy to run on constrained machines, including work laptops where app signing, default file
  associations, native app packaging, and admin friction can become annoying. A browser-first local app,
  served from a local Node process, avoids most of that friction while still preserving the possibility of
  wrapping the same application in Electron later if desired.

  That led to an important architecture decision: the future app should be designed as a single deployable
  local application that runs from Node and serves both the UI and the API. The frontend should not directly
  own filesystem access. Instead, the local server should own file reads, writes, folder browsing, export
  orchestration, and any file-watching behavior. The browser UI should talk to that local server over an
  internal API. This means the future app is not “a website” in the normal sense, but rather a local-first
  application delivered through the browser. The value of that architecture is that it gives the easiest
  path for local use, open source adoption, and work-machine compatibility, while still allowing an Electron
  shell later without creating two separate products.

  We also spent time comparing stack choices for that future browser-first local app. At one point Next.js
  was discussed because it can package frontend and backend logic into a single deployable unit. That part
  was recognized as a valid argument. But after comparing it to a Fastify-plus-vanilla-JS approach, the
  current preference became Fastify plus vanilla HTML/CSS/JS. The reason was that this product is not
  expected to become a giant frontend system with many contributors or extreme UI complexity, and the hard
  problems in this app are not particularly React-specific. The hard parts are markdown rendering, Mermaid,
  local file workflows, export fidelity, and document-oriented UX. For that shape of product, a lightweight
  Fastify server with a simple browser UI feels more honest, lower-overhead, and easier to operate as a
  local utility. The fact that it can still be a single deployable unit was important. So the stack
  direction we settled on is: local Node runtime, Fastify server, vanilla HTML/CSS/JS frontend, TypeScript
  for core and server logic, shared reusable render/export domain code, and possibly an Electron wrapper
  later.

  That last point about shared code is important. Even though the current Electron app is now treated as a
  prototype, we do not intend to throw away all of its internals. The shared core logic in the current
  prototype is considered valuable: markdown rendering, Mermaid handling, export preparation, warning
  generation, and document normalization are all useful seeds for the future architecture. The long-term
  plan is to preserve the reusable document/render/export logic conceptually as a shared core layer, while
  moving the application shell and runtime model to the new browser-first local architecture. In the future
  shape, the server would use that core to perform render/export work, and the browser UI would consume the
  outputs via API responses.

  On the product side, we clarified that the app’s purpose is not primarily “a beautiful markdown editor” or
  “a publishing system.” The strongest product story is as a local-first markdown workspace. The main user
  is a technically capable developer, tech lead, architect, or markdown-heavy knowledge worker who works
  with local markdown files in repos or directories and wants a fast way to browse them, read them rendered,
  make light edits, render Mermaid correctly, and export to shareable formats. The product’s real value is
  in that practical workflow. The differentiators we kept coming back to were local file/folder workflow,
  multiple open documents, Mermaid support, and useful export behavior. The product should feel like a
  practical markdown workstation, not a generic note app.

  We then started turning that into planning artifacts. A preliminary architecture overview was written to
  capture the recommended shape of the future application at a 20–25k-foot level. That document is
  inventory is to make sure future design work covers not just the happy path, but the actual important
  screens and states: initial empty workspace, main reading view, edit mode, lots of tabs, folder-tree-heavy
  workspaces, warning-heavy content, export flows, unsaved-change prompts, external file change conflict
  flows, and narrow-window layouts. The idea is that before formal epics and detailed tech design are
  created, the product should be visualized well enough that the team understands the shell, hierarchy, edge
  states, and workflow transitions.

  So where we are now is this: we have a working Electron prototype preserved under first-pass-poc/, a
  cleaned-up codebase that remains usable, a repo structure that clearly separates the prototype from the
  future build, and a set of preliminary planning docs describing the desired direction. Where we decided we
  wanted to go is toward a browser-first, local-first Markdown Viewer built as a single local Node
  application using Fastify and vanilla JS, with server-owned filesystem and export logic, shared reusable
  core rendering/export ideas, and the option to wrap it in Electron later if needed. The immediate job for
