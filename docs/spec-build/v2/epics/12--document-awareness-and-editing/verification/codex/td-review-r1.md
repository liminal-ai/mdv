# Technical Design Review — Epic 12 (Round 1)

## Verdict

Not ready for downstream story publishing.

## Sequential Read Reflections

1. `docs/spec-build/v2/prd.md`: Epic 12 has to stay local-first, feature-gated, single-document in scope, and conversation persistence is a core acceptance criterion, not optional polish.

2. `docs/spec-build/v2/technical-architecture.md`: The design is constrained by server-owned filesystem/provider lifecycle, canonical workspace identity keyed to source paths rather than temp dirs, additive WebSocket contracts, and coarse-grained script capabilities instead of raw file access.

3. `docs/spec-build/v1/epics/01--app-shell-and-workspace-browsing/epic.md` and `tech-design.md`: Foundational design quality here means explicit contracts, explicit module ownership, and exact AC/TC/test traceability rather than “we’ll figure it out in implementation.”

4. `docs/spec-build/v1/epics/06--hardening-and-electron-wrapper/epic.md` and `tech-design.md`: Later designs are expected to document deviations explicitly, preserve existing architecture, and keep persistence/lifecycle behavior coherent across transient runtime states.

5. `docs/spec-build/v2/epics/10--chat-plumbing/*`: Epic 12 extends a specific chat architecture: separate `/ws/chat`, `ChatWsClient`, `ChatStateStore`, Zod-based message unions, and a per-invocation provider model where `ProviderManager` already owns `sessionId`/`--resume`.

6. `docs/spec-build/v2/epics/11--chat-rendering-and-polish/epic.md`: Epic 12 inherits a client-side chat rendering pipeline, panel toggle/persistence, and polished chat UI. It should not silently rework those seams.

7. `docs/spec-build/v2/epics/12--document-awareness-and-editing/epic.md`: The real obligations are broader than “document-aware chat”: deterministic context injection, dirty-tab safety via Epic 5’s conflict path, canonical workspace identity, persisted CLI session continuity, `chat:file-created`, `chat:conversation-load`, and replace semantics on workspace swap.

8. `ls-tech-design/SKILL.md`: The right review bar here is downstream-consumer validity: every TC needs a real test home, every interface must be concrete, and missing cross-module triggers are design defects, not implementation details.

## Findings

### Critical

- The conversation-swap protocol is incomplete, and the current design cannot reliably replace the chat state when the workspace changes to one with no saved conversation. The epic requires workspace switches to replace the conversation with the new workspace’s messages or an empty state (`docs/spec-build/v2/epics/12--document-awareness-and-editing/epic.md:311-318`), and `chat:conversation-load` is defined with replace semantics (`docs/spec-build/v2/epics/12--document-awareness-and-editing/epic.md:489-492`). But the top-level design only extends client→server chat traffic with `chat:send` context (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design.md:337-342`), while the server companion assumes `ws-chat.ts` somehow “coordinates” workspace changes (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-server.md:489-500`) without defining any trigger or message for that event. Worse, the connect path only sends `chat:conversation-load` if a persisted conversation exists (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-server.md:804-813`), and the test plan hard-codes “No conversation-load for empty workspace” (`docs/spec-build/v2/epics/12--document-awareness-and-editing/test-plan.md:249-250`). On a switch from workspace A to workspace B with no saved conversation, the client has no protocol-level signal telling it to clear A’s messages, so stale conversation state survives the swap.

### Major

- The clear-conversation ACs do not have an end-to-end design. Epic 12 requires `chat:clear` to clear persisted messages and the stored CLI session ID (`docs/spec-build/v2/epics/12--document-awareness-and-editing/epic.md:345-374`). The server companion defines `ConversationService.clear()` (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-server.md:419-435`), but the `ws-chat` extension section only shows `chat:send` handling and `chat:done` persistence (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-server.md:817-928`); there is no Epic-12-level `chat:clear` flow that calls both `conversationService.clear()` and the provider reset path. The client companion likewise adds handlers only for `chat:file-created` and `chat:conversation-load` (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-client.md:753-813`) and never upgrades Epic 10’s local-only clear behavior to cover persisted state. The test plan then maps TC-3.3c / TC-3.5a / TC-3.5b only to `ConversationService.clear()` unit tests (`docs/spec-build/v2/epics/12--document-awareness-and-editing/test-plan.md:231-238`), which does not prove the actual user-visible `chat:clear` round trip.

- The truncation-indicator design contradicts the epic and breaks deterministic TC coverage. The epic requires the indicator to show truncation when the active document was actually truncated by the server’s token budget (`docs/spec-build/v2/epics/12--document-awareness-and-editing/epic.md:166-169`). The server companion makes truncation a server-side decision (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-server.md:65-118`, `docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-server.md:127-153`). But the client companion explicitly admits there is no server-to-client contract for truncation status and replaces it with a local heuristic based on content length or file-tree metadata (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-client.md:158-177`). The top-level external contracts add no truncation field or message (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design.md:337-348`), and the test plan only verifies `contextIndicator.update("spec.md", true)` in isolation (`docs/spec-build/v2/epics/12--document-awareness-and-editing/test-plan.md:178-187`). That proves a widget render path, not the required contract that the indicator reflects the server’s real token-budget decision.

- Session continuity is not internally coherent across Epic 10 and Epic 12, and TC-3.6c has no real implementation home. Epic 10 already defines `ProviderManager` as owning `sessionId` internally and extracting it from `StreamParser.parse(event)` result events (`docs/spec-build/v2/epics/10--chat-plumbing/tech-design-server.md:523-557`, `docs/spec-build/v2/epics/10--chat-plumbing/tech-design-server.md:748-769`, `docs/spec-build/v2/epics/10--chat-plumbing/tech-design-server.md:905-952`). Epic 12 then redesigns the seam: `send()` now accepts `systemPrompt` and `sessionId` from the route (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-server.md:876-900`), while the same document later invents a new parser-emitted `sessionId` event and `onSessionId()` API (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-server.md:964-994`) that do not match the Epic 10 parser contract. The route snippets also never show the `onSessionId()` handler being wired before `conversationService.updateSessionId()` is expected to happen. On top of that, TC-3.6c requires recovery when `--resume` references a deleted CLI session (`docs/spec-build/v2/epics/12--document-awareness-and-editing/epic.md:386-389`), but no recovery path is designed; the test plan incorrectly assigns that TC to `conversation.test.ts` (`docs/spec-build/v2/epics/12--document-awareness-and-editing/test-plan.md:241`), even though `ConversationService` cannot observe or remediate CLI `--resume` failures.

- The client companion is not aligned with the actual app architecture it claims to extend. It uses nonexistent store and integration APIs like `store.subscribe('activeTab', ...)`, `store.get('lastRoot')`, and `store.get('activeTab')` (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-client.md:145-155`, `docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-client.md:740-744`, `docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-client.md:774-789`). In the current codebase, `StateStore` only exposes `get(): ClientState` and `subscribe(listener)`; tab identity is `activeTabId`, and root lives under `state.session.lastRoot` (`app/src/client/state.ts:119-139`). The design also proposes dispatching a custom `window` event named `file-changed` to piggyback on file-watch handling (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-client.md:706-721`), but the current app’s external-change path is wired through `wsClient.on('file-change', ...)` in `app.ts`, not through DOM events (`app/src/client/app.ts:1927-1946`). As written, these interfaces are not implementation-ready against the real client architecture.

- The `CONTEXT_READ_FAILED` fallback flow discards session continuity and persistence in a way that conflicts with the epic’s conversation guarantees. In the `chat:send` catch path, the route sends a `CONTEXT_READ_FAILED` error and then falls back to `providerManager.send(msg.messageId, msg.text, buildSystemPrompt(), null)` (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-server.md:859-870`). That bypasses the stored workspace session ID even if one exists, so a follow-up message after a read failure starts a fresh CLI session rather than honoring AC-1.3 / AC-3.3. It also skips the earlier user-message persistence block (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-server.md:841-850`), which means the fallback conversation can diverge from what the user actually sent. The epic only says the developer gets visible feedback if the document cannot be read (`docs/spec-build/v2/epics/12--document-awareness-and-editing/epic.md:398-404`); it does not authorize dropping session continuity or persistence.

### Minor

- Cross-document count and traceability bookkeeping is wrong. The epic contains 18 ACs and 51 unique TCs (`docs/spec-build/v2/epics/12--document-awareness-and-editing/epic.md:107-415`), but the top-level tech design still claims “~57 TCs” (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design.md:22`, `docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design.md:509`) and the test plan says “All 57 TCs from the epic are mapped” (`docs/spec-build/v2/epics/12--document-awareness-and-editing/test-plan.md:414`). The mapping table actually repeats four TC IDs (`TC-1.4c`, `TC-1.5c`, `TC-3.1d`, `TC-3.3d`) rather than mapping 57 distinct conditions. This does not create a missing-TC gap, but it does create exactly the count drift and traceability ambiguity the review was asked to verify.

- Two quantitative claims are internally inconsistent. First, the client companion estimates conversation restoration at `200-500ms` without Mermaid and `300-900ms` with Mermaid (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-client.md:278-289`) and then immediately calls that “well within the 200ms NFR” (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-client.md:291`), which contradicts the epic’s `<200ms` startup-read NFR (`docs/spec-build/v2/epics/12--document-awareness-and-editing/epic.md:615-617`). Second, the top-level token-budget rationale says a 5,000-line spec document is “typically 150-200KB” and “fits within” a 100,000-character budget (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design.md:66-79`), but 150-200KB does not fit inside 100KB. Neither issue is a contract blocker by itself, but both weaken the design’s sizing rationale.

## What Else I Noticed But Chose Not To Report

- The overall chunk breakdown is structurally sensible. Chunk 0 is explicit, and the broad vertical slices are story-shardable once the interface gaps above are resolved.

- The new module names are mostly coherent across the design set. I did not find a broad file-tree/responsibility drift comparable to the issues above.

- A few code snippets omit imports or supporting helpers, but I did not treat those as findings because companion designs often use partial snippets. I only reported omissions where the missing piece changes the contract or makes the flow impossible.

- I noticed some low-level hardening opportunities in the local-link path validation logic, but I left them out because the larger protocol and ownership issues above would cause more immediate downstream churn.
