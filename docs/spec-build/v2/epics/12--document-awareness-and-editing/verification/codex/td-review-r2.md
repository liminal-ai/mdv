# Technical Design Review — Epic 12 (Round 2)

## Verdict

Not yet approved for story publishing.

## Confirmed Fixes

The Round 1 fixes were materially applied in several important places:

- `chat:conversation-load` now has explicit empty-array semantics in the top-level design, server design, and test plan (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design.md:347-349`, `docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-server.md:805-827`, `docs/spec-build/v2/epics/12--document-awareness-and-editing/test-plan.md:247-253`).
- `chat:clear` is now extended server-side to clear persisted state and emit an empty `chat:conversation-load` (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-server.md:1023-1054`).
- `chat:context` is now defined in the external contracts and server schema, and there is a client-side `onContext()` listener path (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design.md:347-349`, `docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-server.md:734-753`, `docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-client.md:138-173`, `docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-client.md:641-689`).
- The top-level and test-plan count bookkeeping is much better: the design now says 51 TCs and 81 tests, and the arithmetic totals reconcile (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design.md:503-510`, `docs/spec-build/v2/epics/12--document-awareness-and-editing/test-plan.md:405-420`).

## Remaining Findings

### Major

- The client-side store/API corrections from M4 are still incomplete, so the design still does not match the real app architecture it claims to extend. The updated client doc now uses `store.subscribe(callback)`, but it still checks `changed.has('activeTabId')` (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-client.md:150-151`, `docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-client.md:805-806`) even though the real `StateStore` passes `changed` as `Array<keyof ClientState>`, not a `Set` (`app/src/client/state.ts:108-124`). It also still reads `state.lastRoot` / `store.get().lastRoot` in multiple places (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-client.md:154-155`, `docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-client.md:503`, `docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-client.md:762-763`, `docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-client.md:797-798`), but the actual client state has `session.lastRoot`, not a top-level `lastRoot` field (`app/src/client/state.ts:87-106`). The reload path also now calls `handleFileChange({ path, event: 'modified' })` (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-client.md:735-738`), but there is no such reusable helper in the current app; the existing file-change logic is inline inside `app.ts`’s `wsClient.on('file-change', ...)` handler (`app/src/client/app.ts:1927-1946`). That means M4 fixed the direction of the problem, but not the actual interfaces.

- The M3 session-ID redesign is still not internally coherent across the updated docs. The server design correctly changes `send()` to accept only `systemPrompt` and introduces `setSessionId()` (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-server.md:921-975`), and the route now calls `setSessionId()` in both the success and context-error paths (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-server.md:862-866`, `docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-server.md:907-914`). But the client-side sequence diagram still shows the old `send(id, msg, systemPrompt, sessionId)` signature (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-client.md:885-889`). More importantly, the server’s “Provider Manager — Session ID Flow” section is still written against an impossible parser contract: it says `stream-parser.ts` emits `sessionId` events (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-server.md:1070-1078`) and then shows `provider-manager.ts` doing `const event = this.streamParser.parse(line)` followed by `if (event?.type === 'sessionId')` (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-server.md:1085-1091`). That still conflicts with Epic 10’s established parser contract, where `StreamParser.parse(event: Record<string, unknown>)` returns `ParsedEvent[]` and `result.sessionId` is surfaced as part of the parsed result array, not as an emitter-side special event (`docs/spec-build/v2/epics/10--chat-plumbing/tech-design-server.md:905-952`). So the headline fix is right, but the detailed session-ID flow is still drifting across documents.

- One of the new test mappings is still at the wrong boundary, so the “all 51 TCs are mapped” claim is weaker than it looks. TC-4.2b is mapped to `tests/client/steward/chat-panel.test.ts` with the assertion “No `mkdir` or `writeFile` for conversations dir” (`docs/spec-build/v2/epics/12--document-awareness-and-editing/test-plan.md:288-293`). But the test plan’s own mock-boundary section says filesystem calls are server-side boundaries (`docs/spec-build/v2/epics/12--document-awareness-and-editing/test-plan.md:13-23`), and a client chat-panel test cannot credibly prove server-side non-creation of conversation files when the flag is disabled. The TC itself is a system-level isolation requirement from the epic (`docs/spec-build/v2/epics/12--document-awareness-and-editing/epic.md:409-415`), so this still needs a server/integration-layer test home rather than a client DOM test.

### Minor

- `chat:context` is now present in the external contracts and detailed server/client docs, but it is still not fully propagated through the top-level module bookkeeping. The top-level external-contract table includes `chat:context` (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design.md:347-349`), and the `ws-chat.ts` row mentions it (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design.md:432`), but the `server/schemas/index.ts` row still lists only `ChatFileCreatedMessage`, `ChatConversationLoadMessage`, and `PersistedConversation` (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design.md:433`), and the client module rows for `chat-panel.ts` / `chat-ws-client.ts` still only mention `chat:file-created` and `chat:conversation-load` (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design.md:436-437`). This is small, but it is exactly the kind of cross-document propagation drift you flagged as the dominant failure mode.

- The restoration-timing clarification is only half-fixed. The client companion now correctly reframes the `<200ms` NFR as applying to synchronous init work before progressive rendering (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design-client.md:272-287`), but the top-level design still says the total estimate is `200-500ms` and then calls that “well within the 200ms NFR” (`docs/spec-build/v2/epics/12--document-awareness-and-editing/tech-design.md:244-249`). The updated rationale should be propagated to the top-level design too.

## Cross-Document Check

- `chat:conversation-load` empty-array semantics: consistent across the top-level design, server schema/route, and test plan.
- `chat:context`: present in the top-level external contracts, server schema, server route, client listener, and ws-chat test plan. The remaining drift is bookkeeping/documentation, not absence of the message itself.
- `setSessionId()`: used in the two `chat:send` code paths shown in the server doc, but still contradicted by the client sequence diagram and by the stale parser/session-flow section.
- Test counts: 81 total tests now reconcile numerically, and the epic/test-plan TC count is correctly 51. The mapping table intentionally repeats some TCs because a few conditions are split across service and integration tests; that is acceptable.

## What I Noticed But Chose Not To Report

- The always-send `chat:conversation-load` fix materially improves the design. That was the biggest structural problem in Round 1, and it now has a coherent contract.
- The `CONTEXT_READ_FAILED` path is better than before: it now persists the user message first and preserves `--resume` continuity.
- I noticed a few low-level path-validation hardening concerns in the local-file-link logic, but I left them out because the remaining cross-document/interface issues above are more important for story publishing readiness.
