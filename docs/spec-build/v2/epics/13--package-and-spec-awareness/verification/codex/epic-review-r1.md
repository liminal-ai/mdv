## Critical

### 1. The epic explicitly claims full Steward-principle coverage of Epic 9, but it does not actually spec chat equivalents for several Epic 9 operations

**Location:** Epic 13 says package operations through chat apply the Steward principle to "all Epic 9 UI operations" ([epic.md:41](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L41), [epic.md:208](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L208), [epic.md:214](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L214)). Epic 9's actual UI surface includes opening packages, replacing the current package, switching back to folder mode, no-manifest fallback behavior, and extracted-package stale-state behavior ([epic.md:38](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/09--package-viewer-integration/epic.md#L38), [epic.md:40](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/09--package-viewer-integration/epic.md#L40), [epic.md:44](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/09--package-viewer-integration/epic.md#L44), [epic.md:45](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/09--package-viewer-integration/epic.md#L45), [epic.md:72](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/09--package-viewer-integration/epic.md#L72), [epic.md:257](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/09--package-viewer-integration/epic.md#L257), [epic.md:494](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/09--package-viewer-integration/epic.md#L494)).

**Why it matters:** The draft only specifies chat contracts for manifest modification, export, file creation/editing, and viewer navigation ([epic.md:227](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L227)-[epic.md:346](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L346)). There is no chat-side operation for opening a `.mpk`/`.mpkz`, replacing the currently open package, switching from an extracted package back to a folder workspace, or repairing fallback mode in an extracted package. That makes the central Steward-principle claim false relative to the upstream product principle ("anything you can do in the viewer, you can do through the chat" in [prd.md:134](/Users/leemoore/code/md-viewer/docs/spec-build/v2/prd.md#L134)) and leaves downstream design without contracts for core workspace transitions.

### 2. The draft has no coherent model for directory-mode packages, even though Epic 9 depends on one

**Location:** Epic 13 treats workspace type as only `'folder' | 'package'`, with `package` data present only "when a package is open" ([epic.md:609](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L609)-[epic.md:633](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L633)). But the same draft also says `createPackage()` works from folder mode and switches the sidebar into package mode ([epic.md:45](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L45), [epic.md:502](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L502)-[epic.md:526](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L526)). Epic 9 explicitly defines that result as a directory-mode package, not a plain folder ([epic.md:41](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/09--package-viewer-integration/epic.md#L41), [epic.md:264](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/09--package-viewer-integration/epic.md#L264)-[epic.md:277](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/09--package-viewer-integration/epic.md#L277)).

**Why it matters:** After `createPackage()` on a folder, the draft does not say whether subsequent context injection, manifest access, spec metadata, or phase detection should treat that workspace as a `folder` or as a `package`. This is not cosmetic. It directly affects whether `package.navigation` exists, whether `getPackageManifest()`/`updateManifest()` are legal, what the context indicator shows, and how later messages behave. The main happy path of "create a package from this folder, then keep working in chat" is therefore under-specified.

## Major

### 3. Package-aware context extension has no budget/truncation contract analogous to Epic 12's document-awareness contract

**Location:** The epic assumes the manifest will fit in context ([epic.md:68](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L68)) and then adds arbitrary multi-file reads during a single response ([epic.md:151](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L151)-[epic.md:160](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L160)), but punts the actual budget behavior to a tech-design question ([epic.md:839](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L839)). Epic 12 did not do that: it made truncation explicit in scope, ACs, and the `ProviderContext` contract ([epic.md:48](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/12--document-awareness-and-editing/epic.md#L48), [epic.md:156](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/12--document-awareness-and-editing/epic.md#L156)-[epic.md:169](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/12--document-awareness-and-editing/epic.md#L169), [epic.md:457](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/12--document-awareness-and-editing/epic.md#L457)-[epic.md:459](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/12--document-awareness-and-editing/epic.md#L459)).

**Why it matters:** This is user-visible behavior, not just an implementation detail. Without a contract for when multi-file reads are truncated, rejected, or summarized, tests become non-deterministic and the Tech Lead has to invent product behavior that should have been decided here.

### 4. The phase-detection logic is internally inconsistent

**Location:** AC-6.1c says a package with PRD, epic, tech design, and story artifacts should yield detected phase `implementation` ([epic.md:446](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L446)-[epic.md:449](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L449)). But the artifact-detection table later says `implementation` is "not detected from files" and is inferred only when stories exist **and** `specPhase` is `implementation` ([epic.md:783](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L783)-[epic.md:789](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L789)).

**Why it matters:** Those two rules cannot both be true. One says implementation is artifact-derived; the other says it is not. That gives design and verification different targets for one of the epic's core new capabilities.

**Why it matters:** This AC is not testable from the published contract. Right now "instructions" could mean system prompt text, a structured field, or an implementation-only convention. The spec needs to choose the observable contract boundary, not rely on downstream prompt-shaping guesswork.

**Location:** The Feature Overview ends with "All of this also works on regular folders, not just packages" ([epic.md:26](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L26)-[epic.md:27](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L27)). ### 7. The new package-change real-time contract is too incomplete for the UI states this epic introduces

**Location:** The flow says workspace-changing package operations notify the client via `chat:package-changed` ([epic.md:214](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L214)-[epic.md:223](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L223)). The message contract only supports `created`, `exported`, and `manifest-updated` ([epic.md:653](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L653)-[epic.md:679](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L679)). The same draft separately leaves stale-indicator interaction as an open question ([epic.md:844](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L844)).

**Why it matters:** For a streaming UI contract, event coverage and ordering are part of the product behavior. The draft does not define how `chat:package-changed` sequences with `chat:file-created`/`chat:done`, and it has no event for stale-state updates, fallback-to-package transitions in extracted packages, or any future open/switch package operation that the scope claims to support. That leaves the client behavior underspecified exactly where it needs deterministic triggers.

### 8. The script surface drifts away from the technical architecture's "coarse-grained product capability" rule

**Location:** Epic 13 introduces `getFileContent`, `addFile`, and `editFile` as general workspace-file primitives and explicitly assumes the CLI's built-in Read tool may also operate on the workspace as a fallback ([epic.md:70](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L70), [epic.md:711](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L711)-[epic.md:733](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L733)). The technical architecture explicitly recommends the opposite: expose coarse product actions and avoid broad filesystem methods ([technical-architecture.md:353](/Users/leemoore/code/md-viewer/docs/spec-build/v2/technical-architecture.md#L353)-[technical-architecture.md:374](/Users/leemoore/code/md-viewer/docs/spec-build/v2/technical-architecture.md#L374)).

**Why it matters:** This is architectural drift, not just naming. Broad read/write methods enlarge the safety surface, increase provider coupling to filesystem layout, and weaken the product-level abstraction the architecture was trying to preserve. If the epic intends to overturn that principle, it needs to say so directly and justify it.

### 9. The Epic 12 canonical-identity/persistence contract is not extended for the new folder-to-package transitions this draft adds

Epic 13 introduces `createPackage()` on the current folder and richer workspace/package context, but never states whether creating a directory-mode package keeps the existing folder conversation/session ID, changes the workspace identity, or triggers any `chat:conversation-load` behavior ([epic.md:502](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L502)-[epic.md:526](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L526), [epic.md:609](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L609)-[epic.md:633](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L633)).
