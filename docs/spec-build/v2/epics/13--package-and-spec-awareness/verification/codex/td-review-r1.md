# Technical Design Review R1

Epic 13 review against the PRD, v2 architecture, Epics 1/6/10/12, the Epic 13 spec, the available `ls-tech-design` skill, and the four Epic 13 design docs.

I verified the epic still states 28 ACs / 90 TCs, and the design repeatedly claims a 128-test plan. The 128 total is internally repeated consistently, but several supporting counts and interfaces are not.

## Critical

   - But `buildPackageContext()` reads `manifest.rawContent` in [tech-design-server.md](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/tech-design-server.md#L174), while Epic 9’s `PackageManifestResponse` exposes `raw`, not `rawContent`, in [epic.md](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/09--package-viewer-integration/epic.md#L628) and [epic.md](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/09--package-viewer-integration/epic.md#L631).
   - `updateManifest()` and `createPackage()` call `packageService.getManifestFilename()`, `packageService.reloadManifest()`, and `packageService.activateDirectoryPackage()` in [tech-design-server.md](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/tech-design-server.md#L913) through [tech-design-server.md](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/tech-design-server.md#L945), but those methods are not part of the Epic 9 PackageService contract, which only defines `getManifest()`, `create(rootDir, overwrite?)`, and `export(outputPath, compress?, sourceDir?)` in [tech-design-server.md](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/09--package-viewer-integration/tech-design-server.md#L407) through [tech-design-server.md](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/09--package-viewer-integration/tech-design-server.md#L433).

   - This breaks the claimed extension path for AC-3.1, AC-3.2, AC-5.1, AC-5.2, AC-7.1, and AC-7.2.

2. **The script-execution wiring has no viable route-to-provider interface.**
   - The route tries to call `scriptExecutor.setContext(scriptContext)` in [tech-design-server.md](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/tech-design-server.md#L1110).

## Major

1. **The extracted-package fallback repair flow contradicts both Epic 13 and Epic 9 behavior.**
   - Epic 13 requires `createPackage()` to repair missing/unreadable manifests in extracted packages while staying in extracted mode and marking stale in [epic.md](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L592) through [epic.md](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/epic.md#L599).
   - The implementation delegates to `PackageService.create()` in [tech-design-server.md](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/tech-design-server.md#L933) through [tech-design-server.md](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/tech-design-server.md#L945).
   - Epic 9 explicitly says that `create()` "operates in directory mode" and updates package state to `mode: 'directory'` in [tech-design-server.md](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/09--package-viewer-integration/tech-design-server.md#L806) through [tech-design-server.md](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/09--package-viewer-integration/tech-design-server.md#L832).
   - So the specified implementation would convert an extracted fallback package into a directory-mode package rooted at the temp dir, which is the opposite of TC-7.1d/e.

   - But `buildWorkspaceBlock()` only serializes `type`, `title`, `mode`, `manifestStatus`, and `fileCount` in [tech-design-server.md](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/tech-design-server.md#L463) through [tech-design-server.md](/Users/leemoore/code/md-viewer/docs/spec-build/v2/epics/13--package-and-spec-awareness/tech-design-server.md#L518).
   - That means the runtime prompt no longer matches the declared provider context, and the export-path rationale is unsupported by the actual interface.
