• Yes, I think that’s a pretty strong framing.

  React often compresses raw UI complexity into human-friendly abstractions:

  - component hierarchy
  - props
  - local state
  - declarative rerendering
  - reusable patterns

  That is often great for humans because it reduces the need to manually track DOM mutation and wiring.

  But the tradeoff is that some complexity gets re-encoded as indirection:

  - render/effect boundaries
  - hook rules
  - reactive update paths
  - state ownership split across component tree, context, reducers, custom hooks
  - lifecycle behavior that is obvious only if you mentally simulate React

  Models can work with that, but it’s a less direct substrate.

  By contrast, plain JS/DOM code often leaves complexity in a more mechanical, explicit form:

  - this event calls this function
  - this function updates this state
  - this state causes this DOM patch or mount path
  - this workflow happens in this order

  That can be uglier for humans, but often easier for a model to trace because less is hidden behind framework semantics.

  So I’d phrase your point like this:

  - React improves ergonomic compressibility for humans.
  - Explicit JS/DOM often improves traceability for models.

  sprawl.

  The sweet spot for a model-heavy codebase is usually:

  - explicit commands
  - explicit state transitions
  - explicit coordinator modules
  - thin views
  - minimal lifecycle magic

  You can do that in vanilla TS very well.
  You can also do it in React, but only if you are disciplined enough not to let the important behavior dissolve into hooks and effects.

  So for mdv, I think your heuristic is good:

  - don’t choose React just because it makes the code look more normal to humans
  - choose the representation that keeps the app’s real workflow complexity most inspectable and manipulable by the systems that will actually maintain it

  - keeping the core orchestration out of any UI framework
  - maybe improving the vanilla structure first
  - only adopting React later if the main pain becomes view composition itself rather than workflow orchestration
