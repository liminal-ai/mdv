System Design Overview

This document describes the high-level architecture of the Project Atlas platform, covering service boundaries, data flow, and key infrastructure decisions.

Architecture Principles

  • Service isolation: each domain owns its data store and exposes a contract-first API
  • Event-driven communication: services communicate through events, not direct calls, except for synchronous query paths
  • Observable by default: all services emit structured telemetry
  • Gradual consistency: eventual consistency is the default; strong consistency is opt-in where business rules require it

Data Flow

Inbound requests enter through the API Gateway, which handles rate limiting, authentication delegation, and request routing. The gateway is stateless and horizontally scalable.

Request Path

  1. Client sends request to API Gateway
  2. Gateway validates token via Auth Service (cached, TTL 5m)
  3. Request routes to target service based on path prefix
  4. Target service processes and emits domain event
  5. Event Bus distributes to subscribers

Deployment Model

All services deploy as containers on EKS. The CI/CD pipeline uses GitHub Actions with environment promotion gates:

stages:
  - build    # lint, test, container build
  - staging  # deploy to staging, integration
  - canary   # 5% traffic, monitor 15m
  - prod     # full rollout w/ auto rollback

Rollback is automatic if the error budget (measured by SLO burn rate) exceeds the threshold within the canary window.
