Deployment Runbook

Standard operating procedures for deploying Project Atlas services to production. This document covers the full lifecycle from staging validation through canary promotion.

Pre-Deploy Checklist

  • All CI checks passing on the release branch
  • Integration test suite green in staging environment
  • Change request approved in the deployment tracker
  • On-call engineer confirmed and available for the deploy window
  • Rollback plan reviewed and tested within the last 7 days

Deployment Stages

Each service follows a four-stage promotion model. The canary phase is mandatory and cannot be skipped without VP-level approval.

Stage 1: Build & Validate

# Trigger the release build
make release SERVICE=payments TAG=v2.14.0

# Verify the container image
docker inspect atlas/payments:v2.14.0 | jq '.Config.Labels'

Stage 2: Staging Deploy

Deploy to the staging cluster and run the full integration suite. Staging must mirror production configuration within a 48-hour drift window.

Stage 3: Canary Promotion

Shift 5% of production traffic to the new version. Monitor the following metrics for a minimum of 15 minutes:

MetricThresholdAlert Channel
Error rate (5xx)< 0.1%#deploy-alerts
P99 latency< 500ms#deploy-alerts
SLO burn rate< 2x baseline#sre-oncall
Memory usage delta< 15%#infra-alerts

Stage 4: Full Rollout

If all canary metrics remain healthy, promote to 100% traffic. The deployment controller handles this automatically after the canary window closes.

If any metric breaches its threshold during canary, automatic rollback initiates within 60 seconds. See runbooks/incident-response-playbook.md for escalation procedures.

Rollback Procedure

  1. Trigger immediate rollback: make rollback SERVICE=payments
  2. Verify traffic has shifted back to the previous version
  3. Open an incident ticket with deployment logs attached
  4. Notify the #deploy-alerts channel with a brief summary
