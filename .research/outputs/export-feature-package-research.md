# Export Feature - Package & Capability Research

**Date:** 2026-03-20
**Scope:** Package versions, capabilities, and integration details for PDF/DOCX export and Mermaid SSR

## 1. Puppeteer

### Version & Maintenance
- **Latest stable:** `24.40.0` (released 2026-03-19)
- **License:** Apache-2.0
- **Weekly downloads:** 7.3M
- **Actively maintained:** Yes, releases every ~1 week, maintained by Google/Chromium team
- **Repository:** https://github.com/puppeteer/puppeteer

### Module Format
- **Package type:** `"type": "commonjs"` in package.json
- **Dual build:** Both ESM and CJS are supported via conditional exports
  - ESM: `import puppeteer from 'puppeteer'` (resolves to `lib/esm/puppeteer/puppeteer.js`)
  - CJS: `require('puppeteer')` (resolves to `lib/cjs/puppeteer/puppeteer.js`)

### Browser Download Size
- **macOS:** ~170 MB (Chrome for Testing)
- **Linux:** ~282 MB
- **Windows:** ~280 MB
- Downloads to `$HOME/.cache/puppeteer` by default
- Also downloads a `chrome-headless-shell` binary (since v21.6.0)
- Can use `puppeteer-core` to skip download and use an existing browser

### `page.pdf()` Options

Full interface:

```typescript
interface PDFOptions {
  displayHeaderFooter?: boolean;   // default: false
  footerTemplate?: string;         // HTML template, same special classes as header
  format?: PaperFormat;            // 'letter' (default), 'A4', 'legal', 'tabloid', 'ledger', 'A0'-'A6'
  headerTemplate?: string;         // HTML template with classes: date, title, url, pageNumber, totalPages
  height?: string | number;        // custom paper height (overridden by format)
  landscape?: boolean;             // default: false
  margin?: {                       // default: no margins
    top?: string | number;
    right?: string | number;
    bottom?: string | number;
    left?: string | number;
  };
  omitBackground?: boolean;        // default: false - hides default white background
  outline?: boolean;               // EXPERIMENTAL - generate document outline, default: false
  pageRanges?: string;             // e.g. '1-5, 8, 11-13', default: '' (all pages)
  path?: string;                   // file path to save to
  preferCSSPageSize?: boolean;     // default: false - honor CSS @page size
  printBackground?: boolean;       // default: false - MUST set true for background colors/images
  scale?: number;                  // 0.1 to 2, default: 1
  tagged?: boolean;                // EXPERIMENTAL - accessible PDF, default: true (since v22)
  timeout?: number;                // ms, default: 30000, 0 to disable
  waitForFonts?: boolean;          // default: true - waits for document.fonts.ready
  width?: string | number;         // custom paper width
}
```

Units accepted for dimensions: `px`, `in`, `cm`, `mm`. Numbers without units default to pixels.

### Key PDF Capabilities
- Print CSS media type by default; call `page.emulateMediaType('screen')` for screen styles
- Header/footer templates support dynamic values via special CSS classes
- Tagged (accessible) PDFs generated by default since v22
- Document outline generation (experimental)
- CSS `@page` size respected when `preferCSSPageSize: true`
- `-webkit-print-color-adjust: exact` CSS property useful for color fidelity

## 2. Playwright

### Version & Maintenance
- **Latest stable:** `1.58.2` (published ~2026-03-06)
- **License:** Apache-2.0
- **Weekly downloads:** High (major Microsoft-backed project)
- **Actively maintained:** Yes, maintained by Microsoft
- **Repository:** https://github.com/microsoft/playwright

### Module Format
- Supports ESM and CJS
- TypeScript definitions included

### Browser Installation
- Can install Chromium only: `npx playwright install chromium`
- Chromium download: ~281 MB (similar to Puppeteer)
- Also supports `--only-shell` for headless-only (smaller)
- Installs three browsers by default (Chromium, Firefox, WebKit); selective install reduces footprint

### `page.pdf()` Options

Playwright's PDF API is nearly identical to Puppeteer's:

```typescript
// Playwright page.pdf() options
{
  headerTemplate?: string;         // HTML template with classes: date, title, url, pageNumber, totalPages
  footerTemplate?: string;         // same format as header
  printBackground?: boolean;       // default: false
  landscape?: boolean;             // default: false
  pageRanges?: string;             // e.g. '1-5, 8, 11-13'
  format?: string;                 // 'Letter' (default), 'A4', 'Legal', 'Tabloid', 'Ledger', 'A0'-'A6'
  width?: string | number;
  height?: string | number;
  margin?: { top, right, bottom, left }; // defaults to 0
  preferCSSPageSize?: boolean;     // default: false
  tagged?: boolean;                // default: false (NOTE: differs from Puppeteer which defaults true)
  outline?: boolean;               // default: false
  path?: string;                   // file path
  scale?: number;                  // 0.1 to 2
}
```

### Puppeteer vs Playwright for PDF Generation - Comparison

| Aspect | Puppeteer | Playwright |
|--------|-----------|------------|
| `page.pdf()` options | Comprehensive | Nearly identical |
| Default tagged PDF | `true` (since v22) | `false` |
| `waitForFonts` option | Yes | Not documented separately |
| `timeout` option | Yes (30s default) | Handled via general timeout |
| Browser engines | Chrome, Firefox | Chromium, Firefox, WebKit |
| Install size (Chromium) | ~170 MB macOS | ~281 MB macOS |
| `puppeteer-core` option | Yes (skip browser download) | No equivalent lightweight package |
| Headless shell | Separate smaller binary | `--only-shell` flag |

**Recommendation for this project:** Puppeteer is the better choice for PDF export because:
1. Smaller download footprint on macOS (~170 MB vs ~281 MB)
2. `puppeteer-core` allows using an already-downloaded browser (relevant if mermaid-cli also uses Puppeteer)
3. Tagged PDFs enabled by default
4. `waitForFonts` option built in
5. Mermaid CLI already depends on Puppeteer, so sharing the browser avoids duplicate downloads

## 3. docx (by dolanmiu)

### Version & Maintenance
- **Latest stable:** `9.6.1` (released 2026-03-10)
- **License:** MIT
- **Weekly downloads:** 3.6M
- **Actively maintained:** Yes, regular releases, active issue tracker
- **Repository:** https://github.com/dolanmiu/docx
- **Homepage:** https://docx.js.org

### Module Format
- **Package type:** `"type": "module"` (ESM-first)
- **Dual build:** Both ESM and CJS supported via conditional exports
  - ESM: `import { Document, Packer, Paragraph } from "docx"` (resolves to `dist/index.mjs`)
  - CJS: `require("docx")` (resolves to `dist/index.cjs`)
  - UMD: `dist/index.umd.cjs` available
- **TypeScript:** Full type definitions included (`dist/index.d.ts`)

### Capabilities

**Headings:**
```typescript
import { Document, Paragraph, HeadingLevel } from "docx";
new Paragraph({ text: "Title", heading: HeadingLevel.HEADING_1 });
// Supports HEADING_1 through HEADING_6, plus TITLE
```

**Styled Text (TextRun):**
```typescript
import { TextRun, UnderlineType } from "docx";
new TextRun({ text: "styled", bold: true, italics: true, color: "FF0000", size: 28, font: "Arial" });
new TextRun({ text: "underlined", underline: { type: UnderlineType.SINGLE, color: "990000" } });
```

**Tables:**
```typescript
import { Table, TableRow, TableCell, Paragraph } from "docx";
new Table({
  rows: [
    new TableRow({
      children: [
        new TableCell({ children: [new Paragraph("Cell 1")] }),
        new TableCell({ children: [new Paragraph("Cell 2")] }),
      ],
    }),
  ],
});
// Supports nested tables, cell merging, borders, widths
```

**Images:**
```typescript
import { ImageRun } from "docx";
new ImageRun({
  type: "png",  // also: jpeg, gif, bmp
  data: fs.readFileSync("./image.png"),  // Buffer or base64
  transformation: { width: 200, height: 200 },
});
```

**Hyperlinks:**
```typescript
import { ExternalHyperlink, TextRun } from "docx";
new ExternalHyperlink({
  children: [new TextRun({ text: "Click here", style: "Hyperlink" })],
  link: "https://example.com",
});
```

**Other capabilities:**
- Bullet and numbered lists
- Table of Contents
- Headers and footers (per section)
- Page breaks
- Document patching (modify existing .docx files)
- Sections with different page layouts
- Alignment, spacing, indentation
- `Packer.toBuffer()` for generating the final .docx as a Buffer

### Key Assessment
This is a mature, well-maintained library with excellent TypeScript support. It builds DOCX files programmatically from scratch -- it does NOT convert HTML to DOCX. For our use case, converting rendered markdown to DOCX would require building a custom AST-to-DOCX mapper (walking parsed markdown and constructing docx elements).

## 4. HTML-to-DOCX Libraries

### html-to-docx (privateOmega) - Original
- **Version:** `1.8.0`
- **Weekly downloads:** 103.2K
- **License:** MIT
- **Maintenance:** Effectively unmaintained. Last meaningful update was 2023. 97 open issues, 9 open PRs.
- **Quality:** Known issues with Google Docs compatibility, limited CSS support, no ESM exports
- **Verdict:** Not recommended due to maintenance status

### @turbodocx/html-to-docx (TurboDocx fork)
- **Version:** `1.20.1` (published 2026-02-05)
- **Weekly downloads:** 25.6K
- **License:** MIT
- **Maintenance:** Actively maintained fork of the original, backed by TurboDocx Inc.
- **Capabilities:** HTML to DOCX conversion, RTL support, improved compatibility
- **Verdict:** Best option if you want an HTML-to-DOCX path. Actively maintained.

### @packback/html-to-docx
- **Version:** `1.4.13` (published 2026-01-27)
- **Weekly downloads:** 4.2K
- **License:** MIT
- **Maintenance:** Actively maintained
- **Capabilities:** Library-agnostic (accepts DOM Document), works in Node via JSDOM, citation formatting (APA, MLA, Chicago)
- **Verdict:** Niche use case, less battle-tested

### Assessment for DOCX Export Strategy

Two viable approaches:

1. **HTML-to-DOCX conversion** via `@turbodocx/html-to-docx`: Feed rendered HTML into the converter. Simpler pipeline but less control over DOCX styling. Quality depends on HTML/CSS coverage of the converter.

2. **Programmatic DOCX construction** via `docx` (dolanmiu): Walk the parsed markdown AST and build DOCX elements directly. More work upfront but produces higher quality output with full control over styling, and avoids the HTML-to-DOCX lossy conversion.

## 5. Mermaid Server-Side Rendering

### mermaid (core library)
- **Latest stable:** `11.13.0` (released 2026-03-09)
- **Weekly downloads:** 3.7M
- **License:** MIT
- **Actively maintained:** Yes, by mermaid-js org

### @mermaid-js/mermaid-cli
- **Latest stable:** `11.12.0`
- **Weekly downloads:** 235.7K
- **License:** MIT
- **Dependencies:** `mermaid ^11.0.2`, peer dependency `puppeteer ^23`
- **How it works:** Uses Puppeteer to launch headless Chrome, loads mermaid into the page context, calls `mermaid.render()`, extracts the SVG output
- **Output formats:** SVG, PNG, PDF

### Can Mermaid 11.x render in a Puppeteer page context?

**Yes, absolutely.** This is the standard and recommended approach for server-side Mermaid rendering. The mermaid-cli tool does exactly this:

1. Launch Puppeteer browser
2. Create a page
3. Inject the mermaid library into the page via `<script>` tag or `page.evaluate()`
4. Call `mermaid.initialize()` and `mermaid.render(id, definition)` inside the page context
5. Extract the resulting SVG from the DOM

For our use case (rendering Mermaid diagrams during export), we have two options:

**Option A: Use mermaid-cli as a subprocess**
```bash
npx mmdc -i input.mmd -o output.svg
```
Pro: Simple. Con: Spawns a new browser per invocation, slow for multiple diagrams.

**Option B: Direct Puppeteer page rendering (recommended)**
```javascript
// If we already have a Puppeteer browser open for PDF generation,
// we can reuse it to render Mermaid diagrams:
const page = await browser.newPage();
await page.setContent(`<html><body><script src="mermaid.min.js"></script></body></html>`);
await page.evaluate(async (definition) => {
  mermaid.initialize({ startOnLoad: false, theme: 'default' });
  const { svg } = await mermaid.render('diagram', definition);
  document.body.innerHTML = svg;
}, diagramSource);
const svgContent = await page.$eval('body', el => el.innerHTML);
```
Pro: Reuses existing browser, fast for batch rendering. Con: Must bundle/serve mermaid.min.js.

**Option C: Pure server-side without browser**
There are experimental approaches (svgdom, JSDOM) but they are fragile and not production-ready per mermaid-js issue #6634. The Puppeteer approach remains the canonical solution.

### Key Insight
Since both PDF export and Mermaid SSR need Puppeteer, they can share a single browser instance. Launch once, render all Mermaid diagrams to SVG, inject them into the HTML, then generate the PDF -- all within the same browser session.

## 6. NSSavePanel via osascript

### Can osascript invoke a save dialog?

**Yes.** Both AppleScript and JXA (JavaScript for Automation) support the `choose file name` command from Standard Additions, which presents a native NSSavePanel.

### AppleScript Syntax

```applescript
set outputPath to choose file name ¬
    with prompt "Export document as:" ¬
    default name "document.pdf" ¬
    default location (path to desktop folder)
-- Returns: file "Macintosh HD:Users:lee:Desktop:document.pdf"
```

**Parameters:**
| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `with prompt` | text | `"Specify new file name and location"` | Dialog prompt text |
| `default name` | text | `"untitled"` | Pre-filled filename |
| `default location` | alias | User's Documents folder | Starting directory |

**Result:** Returns a `file` object (HFS path). If user cancels, raises error number -128.

**Invocation from Node.js:**
```javascript
const { execFile } = require('child_process');

function showSaveDialog(defaultName, defaultDir, prompt) {
  const script = `
    set outputPath to choose file name ¬
      with prompt "${prompt}" ¬
      default name "${defaultName}" ¬
      default location POSIX file "${defaultDir}"
    return POSIX path of outputPath
  `;
  return new Promise((resolve, reject) => {
    execFile('osascript', ['-e', script], (err, stdout, stderr) => {
      if (err) {
        if (err.code === 1) reject(new Error('User cancelled'));
        else reject(err);
      } else {
        resolve(stdout.trim());
      }
    });
  });
}
```

### JXA (JavaScript for Automation) Syntax

```javascript
// Run via: osascript -l JavaScript -e '...'
var app = Application.currentApplication();
app.includeStandardAdditions = true;

var path = app.chooseFileName({
    withPrompt: "Export document as:",
    defaultName: "document.pdf",
    defaultLocation: Path("/Users/lee/Desktop")
});
// Returns: Path("/Users/lee/Desktop/document.pdf")
```

**Node.js invocation with JXA:**
```javascript
function showSaveDialogJXA(defaultName, defaultDir, prompt) {
  const script = `
    var app = Application.currentApplication();
    app.includeStandardAdditions = true;
    var path = app.chooseFileName({
      withPrompt: ${JSON.stringify(prompt)},
      defaultName: ${JSON.stringify(defaultName)},
      defaultLocation: Path(${JSON.stringify(defaultDir)})
    });
    path.toString();
  `;
  return new Promise((resolve, reject) => {
    execFile('osascript', ['-l', 'JavaScript', '-e', script], (err, stdout) => {
      if (err) {
        if (err.code === 1) reject(new Error('User cancelled'));
        else reject(err);
      } else {
        resolve(stdout.trim());
      }
    });
  });
}
```

### Important Notes
- The dialog does NOT create the file -- it only returns a chosen path
- If the user picks an existing filename, the OS prompts "Replace?" automatically
- The dialog is a real native macOS NSSavePanel -- looks and behaves like any native Save dialog
- Works without any special entitlements or permissions on macOS
- User cancel raises error code -128 (AppleScript) / exit code 1 (osascript process)
- `POSIX file` converts a POSIX path to an alias; `POSIX path of` converts back
